**CORE_RISKS**
- `Enemy snake AI scope` is the biggest risk: “random appearance, random movement” sounds simple, but collision-safe spawning, path updates, growth, snake-vs-snake interactions, and “eat other snakes if smaller” quickly make logic much more complex than a basic Snake clone.
- `Ambiguous combat rules` can break gameplay: “eat other snakes” does not define whether comparison is by length, score, level, or current segment count; without this, core win/loss logic is underspecified.
- `UI ambition vs simplicity` conflicts: “high-end/premium page design”, “multiple dynamic visual effects”, and “cyberpunk aesthetic” can fight the goal of “simple code,” especially in a greenfield React-only app.
- `Performance/rendering risk` on a `30x30` grid with `3-8` AI snakes, effects, animations, overlays, and frequent game ticks may cause unnecessary re-renders if implemented naively with React state per cell.
- `Persistence ambiguity` risks wrong implementation: “local score recording” and “Top 10 leaderboard” likely imply `localStorage`, but it is unclear whether scores are per browser only, per snake type, global, timestamped, or resettable.
- `Game balance risk` is high: enemy count, speed, growth rate, spawn rate, and combat rules can make the game unfair or trivial without explicit tuning targets.
- `Pause/start timing bugs` are common: unclear whether pause freezes timers, effects, enemy AI, score timing, and keyboard input queues.

- `Rendering approach` is unclear: a `div` grid is simpler, but `canvas` may be better for premium effects and performance. React DOM per-cell rendering may still be fine for `900` cells, but less so with heavy effects.

**ALTERNATIVE_DIRECTIONS**
- `Option A: Pure React + CSS grid`

  - Best for: fast MVP, maintainable code, modest effects.
  - Weakness: premium visual effects and animation richness are more limited.
- `Option B: React UI shell + Canvas game board`
  - Tradeoff: slightly more complex architecture, better runtime performance and effects.
  - Best for: cyberpunk visuals, glow trails, particles, smoother enemy rendering.
  - Weakness: game rendering logic is less “React-idiomatic.”
- `Option C: React + lightweight game engine layer`
  - Tradeoff: best long-term structure by separating engine logic from UI.
  - Best for: extensibility, testing, deterministic mechanics.
  - Weakness: more upfront design work.
- `Option D: Scope down enemy interactions for v1`
  - Example: enemies move randomly and can kill player on contact, but only player can “consume” stunned/smaller enemies under simpler conditions.
  - Tradeoff: less feature-complete, much lower logic complexity.
- `Option E: Two persistence views`
  - `Top 10 leaderboard` plus `full recent history`.
  - Tradeoff: slightly more UI/storage work, much clearer user value than only top 10.
- `Option F: Fixed desktop-first release`
  - Tradeoff: avoids responsive control complexity now.
  - Best for: keyboard Snake gameplay quality.
  - Weakness: weaker phone/tablet experience.
- `Option G: Visual effects tiers`
  - Example: `Low / Standard / Ultra` or `Reduced Motion`.
  - Tradeoff: extra settings UI, much better accessibility/performance flexibility.
